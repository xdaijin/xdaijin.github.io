## 基础知识
### 网络分层模型
#### osi七层模型  
1. 应用层：直接为应用进程服务，如http、ftp、smtp、dns  
2. 表示层：负责数据编码转换，数据压缩
3. 会话层：负责建立、维护、控制会话
4. 传输层：负责分割、传输、组合数据，如tcp、udp
5. 网络层：负责逻辑上的网络地址定位和路由，如ip、arp、icmp、rarp
6. 链路层：负责数据物理上的传输方式和地址定位，差错校验，封装成帧，如ppp、mac、crc
7. 物理层：负责如何在各种物理设备上传输数据

#### tcp/ip模型  
七层模型分层太多，负责和冗余，实施困难，后来进行了简化，保留了四层，上三层和下二层都进行了合并

1. 应用层
2. 传输层
3. 网络层
4. 数据链路层

### 粘包拆包
如果一个数据包的大小如果超过mtu，那协议会将包拆分成多个包依次发送，接收方也要对接收到的包进行组合

如果短时间内发送多个很小的数据包，那协议会将多个包合并成一个包，接受方对收到的包要根据预先定义好的协议栈进行拆包，粘包是negal算法实现的结果

解决办法：将数据分为定长的数据头和不定长的数据体，数据头里面带有表示数据长度的字段

### 拥塞控制
通过控制拥塞窗口(cwnd)和拥塞阈值(ssthresh)的变花来达到对流量的控制，避免数据发送过多导致网络无法承受而产生拥塞,对拥塞控制的描述
> 只要网络中没有出现拥塞，拥塞窗口的值就可以再增大一些，以便把更多的数据包发送出去，但只要网络出现拥塞，拥塞窗口的值就应该减小一些，以减少注入到网络中的数据包数。

本质就是通过控制cwnd的指数增长还是线性增长来控制流量变化，在ssthresh以下就是指数增长，在ssthresh以上就是线性增长。如果发生快速重传就降低ssthresh，如果发生超时重传就重置cwnd

拥塞的判断：发生超时重传或者收到三个重复的ack

拥塞控制算法：
- 慢启动：开始设置cwnd为1，然后按指数增长，直到发生拥塞或者到达阈值

## http
### status code
- 1xx: 信息类状态码
- 2xx: 成功类状态码
- 3xx: 重定向错误码
- 4xx: 客户端错误状态码
- 5xx: 服务端错误状态码

### restful
REST即表述性状态传递（英文：Representational State Transfer，简称REST）。将资源抽象成对象，然后用url来定位，通过get、put、post、delete来实现curd操作的web设计风格

### methods
- get: 获取资源 
- put: 更新资源
- post: 新增资源
- delete: 删除资源
- head: 查询资源元数据
- option: 查看资源支持的方法

### http1.0 1.1 2.0
#### 1.0
1.0 浏览器和服务器只保持短暂的链接，浏览器的每次请求都要和服务器建立一次链接
#### 1.1
- 引入长连接，一个连接能够被多个请求复用
- 同一个连接里面服务器一次只能处理一个请求，必须等上个请求结束才能处理下一个请求，本质只是省去了建立连接的过程
- 新增了option、put、delete方法
- 新增了一些新的请求头和响应头

#### 2.0 
- 采用二进制格式而不是文本格式
- tcp实现多路复用，不需要安装顺序处理和返回结果
- 使用包头压缩，降低传输开销
- 服务器推送

### https
在http的基础上使用ssl进行加密，默认端口是443

https过程
1. 客户端发起请求, 生成随机数发送给服务端，并告诉服务自己支持哪些加密算法
2. 服务端生成随机数, 在双方都支持的密钥加密算法中选择一种，然后将证书、随机数、公钥发送给客户端
3. 客户端验证证书，然后生成密钥，用服务的公钥加密密钥发送给服务端
4. 服务端和客户端用同样的密钥加密数据进行传输
